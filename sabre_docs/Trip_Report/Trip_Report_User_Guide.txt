1.1. Introduction
We address this guide mainly to architects and developers who need a mechanism of generating various lists or reports based on flights and PNRs stored in Sabre Open Systems. Business users can also benefit from this documentation to get familiar with the service capabilities.
This chapter contains general information about the service. In the next chapters, we describe the mechanisms implemented within the service, including all the reports offered and a list of error and warning messages. At the end of the User Guide you can find a glossary of the terms used, as well as how to reach Technical Support, if needed.
1.2. Purpose
Once a PNR has been created on the Sabre Host, the Native Sabre and Trip_ReportsRQ web service offers functionalities to airlines to generate reports. The Trip_ReportsRQ service allows displaying booking, ticketing and inventory data in form of reports for specific flights.
The user can generate reports through Native Sabre commands and via Trip_ReportsRQ API calls.
There are four main groups of reports:
1. Passenger Lists – group of reports that allow displaying passenger listswith specific information about flight and passengers.
2. Inventory Reports – ad hoc reports that allow retrieving details on flights
3. Group / Corporate Name PNR Lists – reports which display large party and group/corporate name PNRs to assist airlines in managing their group products.
4. Reacommodated Passengers – reports that allow retrieving historical reaccommodation information for all passengers on the flight, either for cancelled or active flights.
The next chapters describe reports from each presented group in more details.
1.3. Availability
An authorized airline user can benefit from Trip_ReportsRQ both through the Native Sabre channel (some host commands utilize our API in the background) and the web service one. We offer standard service functionalities to all airline clients.
The service runs in 24x7 mode. In case of any planned maintenance windows, we send the wide standard notifications in advance.

2.2. Trip_ReportsRQ request and response structures
Trip_ReportsRQ request message (RQ)
An overview on the Trip_ReportsRQ service design, with the key XML sections highlighted, is presented below. For more information please refer to next paragraphs. HEADER: It contains e.g. BinarySecurityToken obtained when authorizing user’s web service session. BODY: <Trip_ReportsRQ> required block, some its attributes like Version are required too. <Report> required block <ReportName> required block Each report is defined by ReportName, e.g. MLG, PFX. <ReportFormat> optional block Each report has a specific format defined, e.g. G. <Criteria> required block, includes the following search criteria (some of them are mandatory for some reports): FlightCriteria: AirlineCode, FlightNumber, DepartureAirport, ArrivalAirport, InboundAirport, OutboundAirport, DepartureDateTime, EndDateTime, StatusCode, ClassOfService, Frequency; NameCriteria: LastName (can be used once per request); PosCriteria (can be used once per request); SSRCriteria (can be used once per request); OSICriteria (can be used once per request); DateCriteria: StartDate, EndDate (can be used once per request); BookingCriteria: Station, AgentHomeCity, AgentSine (can be used once per request).
Trip_ReportsRQ response message (RS)
The response is an XML file that contains e.g. list of passengers or details of inventory per flight. Unless there are no PNRs (passengers) meeting the report criteria, an empty list is returned.

2.3. Service Security
The reports’ generation is available only for authorized AS users that provided a valid token in the web service request.
Users can retrieve data only from the partitions they have access to. In a common case it means that a user of given airline cannot access data of any other carrier unless it is allowed in the native Sabre system too.

3.2. MLJ – List of Confirmed Passengers with Married Segments
The MLJ Report returns list of passengers for married segments with record locator and status code for all board points or specific board point.
Key characteristic:
Report returns only confirmed passengers, taking into consideration following status codes only: HK, KK, KL, BK, CG, DK, FG, GK, IG, NK, OB, OS, OX, PS, RR, SC, TK, CH, CR, LK, YG, ZB, ZH, ZK, ZR (Note: all these codes may not be used in all airlines partitions)
1. Data returned in response is ordered by departure point. For example, for a flight operating en route SCL-LIM-LAX, report returns passenger list for SCL-LIM, SCL-LAX and LIM-LAX. Please see Use Case 2 below.
2. Report does not return passenger type Infant.
MLJ Report is available in Native Sabre and Trip_ReportsRQ.
Note: the report requires a dedicated activation.

3.3. PFX – List of Confirmed Passengers with their SSRs/OSIs
The PFX Report presents list of confirmed passengers including their SSRs or OSIs for all board points with the counts for each SSR/OSI type.
Key characteristic:
If Departure Airport and Arrival Airport are specified, report returns data for the leg or legs between the two locations.
1. If Departure Airport and Arrival Airport are not specified, report returns data for all flight legs ordered by departure point. For example, for a flight operating SCL-LIM-LAX, report returns passengers for each leg in the following order: SCL-LIM, SCL-LAX and LIM-LAX.
2. If Departure Airport but not Arrival Airport is specified, report returns data for all legs from the Departure Airport.
3. Report returns all PNRs containing SSRs or OSIs (all or matching specific codes entered) with counts for each.
4. It is possible to search for all SSR/OSI codes or a specific one.
5. If no specific SSR/OSI is included in the request, all SSR/OSI types will be presented.
6. If a specific SSR/OSI is included in the request, no other SSR/OSI types are presented.
7. If a specific SSR/OSI is included in the request, but there are no PNRs matching that criteria, empty list will be presented.
8. If user decides to include the SSR/OSI code in the request, one of the following SSR/OSI types needs to be defined: All, AFX or GFX.
9. It is possible to include up to sixteen specific SSR or OSI codes in a single entry.
Sabre Inc. Confidential / All Rights Reserved Passenger Lists 19
10. The response will present each individual name and the SSRs/OSIs that apply to that name.
11. A summary of the SSR/OSI counts is presented in each response. Counts are calculated for each SSR/OSI code present in the response. These are counts of SSRs/OSI for the specific flight requested. Eg. from Trip_ReportsRS:

3.7. MLBO – List of Confirmed Passengers with an Outbound Connection from Specified Flight
The MLBO Report presents list of confirmed passengers for specific flight, on specific date, which have an outbound connection for the specified flight. In other words, specified segment is not the last one in the itinerary.
Key characteristic:
1. Report returns only confirmed passengers, taking into consideration following status code only:
HK, KK, KL, BK, CG, DK, FG, GK, IG, NK, OB, OS, OX, PS, RR, SC, TK, CH, CR, LK, YG, ZB, ZH, ZK, ZR (Note: some of the codes may not be present for certain airlines)

3. Report does not return passenger type Infant.
MLBO Report is available in Native Sabre and Trip_ReportsRQ.

3.13. TRIN – Inbound Transfer Information List with Names
The TRIN Report presents list of passengers with a connection from the specified flight at the specified city. The list is sorted by class of service on the connecting flight, counts the number of passengers per connecting flight. Report presents passenger names, record locator, class of service and number in party for each transfer passenger additionally to TRI Report.
If the connecting flight has multiple off points, each of them is calculated separately.
Key characteristic:
1. Report returns only confirmed passengers, taking into consideration following status code only:
HK, KK, KL, BK, CG, DK, FG, GK, IG, NK, OB, OS, OX, PS, RR, SC, TK, CH, CR, LK, YG, ZB, ZH, ZK, ZR (Note: some of the codes may not be present for certain airlines)
2. Report does not return passenger type Infant.
TRIN Report is available in Native Sabre and Trip_ReportsRQ.
Note: this report requires a dedicated activation.

4.1. General Information
Inventory reports gather and present flight information by segment, legs, cabin, fare class, segment status, and board point. Reports include booked, ticketed, authorization levels, actual capacity and available to sell counts.
BSF, BSR and BST are available in Native Sabre and via direct Trip_ReportsRQ API calls. BGR Report is accessible through the web service channel only.
All Inventory Reports are available up to 48 hours after the flight departure.
The next sections of this chapter describe the functionality of each Inventory Report in details.
4.2. BSF – Full Flight Booking Report
The BSF Report presents detailed inventory data by class of service for departure cities and flight segments.
BSF Report follows the below rules:
1. If origin and destination is included in the input, the system will determine the flight routing and return the report for the leg(s) and segment(s) included in the request.
2. If no origin and destination is included in the input, the system will determine the flight routing and return the report for all legs and segments in the routing.
Example for routing: SCL-MAD-FRA
Option 1 - Input: BSF/LA507/10DEC/SCLFRA Report on Legs: SCL, MAD Report on Segments: SCL-MAD, SCL-FRA, MAD-FRA
Option 2 - Input: BSF/LA507/10DEC/MADFRA Report on Leg: MAD Report on Segment: MAD-FRA
Option 3 - Input: BSF/LA507/10DEC/SCLMAD Report on Leg: SCL Report on Segment: SCL-MAD
Option 4 - Input: BSF/LA507/10DEC Report on Legs: SCL, MAD Report on Segments: SCL-MAD, SCL-FRA, MAD-FRA
BSF Report is available in Native Sabre and Trip_ReportsRQ up to 48 hours after the departure.
Please note, this report requires a dedicated activation.

6.1. MLR – Passenger Reaccommodation List
The MLR Report presents passenger list of all PNRs that were on a disrupted flight, even if the flight has been cancelled. The MLR report shows also the flight on which the passengers were protected after the reaccommodation process has been completed. The list will also include flights which have undergone a schedule change that did not result in a change of flight, i.e., a time change.
The date used in the input may range between 30 days prior and 330 days after the date the request is made.
You can retrieve the Passenger Reaccommodation List for:
 Any given flight, canceled or active, which displays historical reaccommodation information for all passengers on the flight.
 All passengers that were originally on the requested flight along with the flight to which they were reaccommodated including the carrier code, flight number, date, class of service, departure and arrival cities and status code.
 All passengers that were originally on another flight and reaccommodated to the requested flight, e.g., a request for flight 600/21MAR will include passengers reaccommodated from 600/21MAR to another and passengers reaccommodated to 600/21MAR from another flight.
 If reaccommodation is from a single flight to a connection, the connection will be included in the display.
 If there is no reaccommodation to a different flight, e.g., equipment change or departure/ arrival time changes only, the list include the flight as both the ‘to’ and ‘from’ flight.
 The list for each passenger includes all reaccommodations/flight changes made since the passenger originally booked their flight. This can be referred to as historical reaccomodations.
 If the PNR has been through multiple schedule changes all “FROM” (disrupted flight) and “TO” (protection flight), flights are shown as well. The report will include information from past date PNRs up to 30 days after the last flight in the PNR was flown. E.g.:
On 1 June 2017 John Smith booked space on LA600 SCLLIM for travel on 1 January 2018. Flight 600 was subject to schedule changes on 5 June, 10 July, 1 August, 9 September, 19 October, 22 November and 19 December.
A request made on 10 June 2017 for flight LA602/1JAN will show Mr. Smith as having been impacted on the 5 June schedule change.
A request made on 20 December 2017 for flight 602/1JAN will show Mr. Smith as having been impacted by the 5 June, 10 July, 1 August, 9 September, 19 October, 22 November and 19 December schedule changes.

Providing the support desk with the necessary files at the time of initial contact improves our ability to troubleshoot and provide a timely resolution:
 Please include the Sabre Pseudo City Code (PCC) or related Office Acocunting Codes (OAC) or Domain where the issue is occurring.
 When reporting an issue with SOAP based Sabre APIs the input and output payloads are required. Please attach the payloads as separate files, and name them clearly.
 To help ensure that our environment is free of viruses, our policy mandates that all messages received by Sabre from external sources follow special file name guidelines.  If your correspondence is regarding a previously reported issue, please include the service incident (“SI”) number in the subject line of your message.
 The subject line should contain a brief description of the problem or question. Please avoid a subject line containing only the words ‘Question’, ‘Issue’, ‘Problem’, etc.
 In the body of the e-mail include as much information as possible to explain the problem and the steps necessary to recreate it.
