Overview 1 

1 . 1 . I n t r o d u c t i o n 

 

This guide is for architects and developers to learn how to compose XML formatted requests for 
Trip_SearchRQ service version 4.5.0. The document may be beneficial for business users as it contains 
a couple of service request samples presenting use cases. 

The User Guide describes the Trip_SearchRQ 4.5.0 current functionalities. Some of the search options 
may require dedicated activations, what is mentioned in this specification accordingly. The baseline 
pre-requisite for using Trip_SearchRQ is having PNRs stored in the Trip Services databases. 

 

 

This chapter contains general information about the service. In chapter 2 the available search 
capabilities available, both for the Sabre Airline carriers and Sabre Travel Network users. Chapter 3 
presents the search criteria in more details, with use cases attached. In the next chapter there is a list of 
error and warning messages that Trip_SearchRQ may produce. At the end of the User Guide you can 
find a glossary of terms used, as well as how to reach Technical Support, if needed. 

Both WSDL and XSD files are published separately to the User Guide, although those files are 
complementary to this document. The complete requests and responses are published separately too. 

 

1 . 2 . P u r p o s e 

 

 

Once a PNR has been created on the Sabre Host, the Trip_SearchRQ web service offers functionality to 
allow the Airline or Agency to search for bookings by many different criteria. The service can be used 
as an alternative to host searches available in native Sabre system, however the capabilities of 
Trip_SearchRQ service are a little bit different and more complex. 

Trip_SearchRQ allows you to search for most common data elements that are in a PNR. It is a highly 
efficient solution that is designed for maximum reliability. 

The main goals of Trip_SearchRQ service is to search for the bookings. If you are interested in 
retrieving PNR data only having the PNR locator already known, it might be an option to use 
GetReservationRQ service alternatively.

activate the asynchronous search, please contact your Sabre Representative.

User needs to specify at least one Primary search criteria in the search request. All other criteria are 
named Secondary, and are optional. Trip_SearchRQ call without any Primary criteria will result in 
“There is no primary criteria specified” error response. 

The following is the list of primary search criteria that can be used in the service request: 

• LocatorCriteria/Locator/Id 
• DateCriteria/UpdateDate 
• NameCriteria/LastName 
• TelephoneCriteria/PhoneNumber 
• EmailCriteria/EmailAddress 
• TicketCriteria/TicketNumber 
• EMDCriteria/EMD/EMDNumber/EMDType 
• LoyaltyCriteria/Loyalty/ProgramID 
• ProfileCriteria/Profile/ProfileID 
• FlightCriteria/Flight/DepartureAirport 
• FlightCriteria/Flight/DepartureCountry 
• FlightCriteria/Flight/DepartureCity 
• FlightCriteria/Flight/ArrivalAirport 
• FlightCriteria/Flight/ArrivalCountry 
• FlightCriteria/Flight/ArrivalCity 
• FlightCriteria/Flight/FlightNumber; 
• FlightCriteria/Flight/MarketingFlightNumber; 
• FlightCriteria/Flight/OperatingFLightNumber; 
• FlightCriteria/Flight/DepartureDateTime; 
• FlightCriteria/Flight/AirlineCode 
• FlightCriteria/Flight/MarketingAirlineCode; 
• FlightCriteria/Flight/OperatingAirlineCode; 


 

 

Primary search criteria cannot be used in combination with MatchMode=“ANYWHERE” and “END”. 
If used, following error response will appear: “There is no primary criteria specified”. 


2 . 4 . F i l t e r i n g q u a l i f i e r s 

 

 

C o m b i n i n g m a n y s e a r c h c r i t e r i a a t a t i m e 

Most search criteria in the request payload are in ‘AND’ relation, which means that the PNR response 
will be accordingly narrowed down. The exception to this rule is an option to provide multiple PNR 
record locators at a time that may result in returning many bookings in the response. Please review the 
WSDL and XSD files for more details. 

When a combination of search criteria returns “Too many results” message, it usually means the search 
criteria are too weak. In the service request, it is possible to specify the MaxItemReturned attribute too. 
Given the boundary for maximum number of results reached, the mentioned message occurs. Please 
note that the service engine tries to predict the number of results prior to producing the response. If the 
number of expected results is huge, the service response may contain the indicated warning message 
too. In case of such warning messages, please adjust the request payload. 

 

M a t c h M ode s 

 

For several search criteria MatchMode attribute is defined. It means that searching engine is not 
constrained only to EXACT searches but provides also START, END, ANYWHERE, SIMILAR, 
EXACT_NO_COMPRESS search capabilities as well. There may be a different set of MatchMode 
options for each filtering criteria. 

 

 

MatchMode option 

Description 

EXACT 

EXACT searches for spelling based on compressed names (without 
white spaces) e.g., a search for the LastName Mc Donald would have 
an entry that looks like 

<LastName MatchMode="EXACT">MCDONALD</LastName> 

START 

START would be used to search without spelling the entire word out 

(in Native Sabre this search is using wildcard characters) e.g., 

<LastName MatchMode="START">SM</LastName> 

END 

END can return a very large amount of data which will cause the return 

to error. It can be used as a secondary condition to limit the number of 
responses e.g., searching for the names SMYTH/ME 
(SMYTH/FIRSTNAME, SMYTH/KIME, SMYTH/NAME etc) 

<FirstName MatchMode="END">ME</FirstName> 

<LastName MatchMode="EXACT">SMYTH</LastName> 

ANYWHERE 

ANYWHERE can return a very large amount of data which will cause 

the return to error. It can be used as a secondary condition to limit the 
number of responses e.g., searching for the names SMYTH/ME 
(SMYTH/FIRSTNAME, SMYTH/NAME, SMYTH/MENU, 

SMYTH/JAMES etc.). MatchMode ANYWHERE cannot be used in 
combination with Primary Search Criteria.

DateTime format is using the following format specified by ISO 8601 

YYYY-MM-DDThh:mm:ss 

with time values using the 24-hour clock (e.g. 20 November 2003, 1:59:38 pm UTC becomes 2003-11- 
20T13:59:38Z). Time format follows the pattern HH:MM:SS 

“After”, “Before” and “Is” match modes accept DateTime format. “TimeAfter”, “TimeBefore” and 
“TimeIs” accept Time format only. In case of the “Is” attribute the time isn’t taken into consideration in 
search logic because the time is truncated.

2 . 8 . P a s t D a t e P N R s e a r c h e s 

 

 

The Past Date PNR searches allow to look for PNRs which are no longer available to users and have 
been purged from the system. 

Past Date PNR search is a standard offering for travel agencies (TN) and does not require additional 
activation. 

This functionality is made available to all airlines however, it is not a standard offering and requires 
customer activation. 

The Past Date PNR search is narrowed down using SearchType=”INACTIVE” option present in the 
ReturnOptions tag of the Trip_SearchRQ. You can specify whether the search should be done against 
active or Past Date PNRs. 

The allowed values for the SearchType option within the ReturnOptions Trip_SearchRQ tag are 
presented in the following table. This attribute is optional; if provided, a single value can be assigned. 

 

 

Search Type 

Definition 

ACTIVE

2 . 9 . A s y n c h r o n o u s s e a r c h e s 

 

 

The asynchronous searches are dedicated for the long running searches that in normal (synchronous) 
mode could result in timeouts. The asynchronous search mode is enabled on search view configuration 
level. Although the asynchronous searches support long running queries, there is a limit on the number 
of such queries invoked per hour. 

All the standard views are not asynchronous; however, it is possible to request an asynchronous one as 
a special service. If you are interested in such kind of searches please contact Technical Support / 
Development Team. 

The samples / use cases presented below uses the following keywords: 

• <async_search_view> - search view name with enabled async search option. For async 
mode there is also possibility to enable pagination of the result, it is enabled from search 
view configuration level. 
• PageSize attribute from ReturnOptions – it allows control of the size of page in case of 
pagination enabled, it is optional and default size could be customized on search view level 
by Dev team. The pagination is supported for all asynchronous views and only for 
asynchronous views.

2 . 1 0 . S e r v i c e S e c u r i t y 

 

 

The PNR search options are available only to authorized users that provided a valid token in the web 
service request. For most service views, the Trip_SearchRQ supports branch access and global branch 
access rules in a consistent manner as the native Sabre application. 


Users can retrieve data only from the partitions and/or pseudos they have access to. In a common case it 
means that a user of given Airline cannot access data of any other carrier unless it is allowed in the 
native Sabre system. 

There are viewership rules implemented in the service, although Trip_SearchRQ does not inherit the 
viewership rules from the native Sabre application automatically. 

The authorization based on OACs (Office Accounting Codes) is not supported with the service. 

 

2 . 1 1 . P e r f o r m a n c e 


 

 

Search criteria defined in this User Guide should work in less than a few seconds. For the search 
queries that lasts a longer than 20 seconds, the queries will be cancelled, and a timeout will be returned 
to the client. 

For those clients that require long running queries we have a semi real time request, token-based option 
(see a section describing asynchronous searches). Please contact out Technical Support if you found 
any performance issues with the service.

3 . 1 . T r i p _ S e a r c h R Q m a i n b l o c k 


 

 

Elements: 

ClientContext, ReadRequests 

Attributes: 

ConversationID; Compression; EchoToken; TimeStamp; Target; 
Version; SequenceNumber; PrinaryLangID; AltLangID; ReturnContent 

• ConversationID - is a string identifying the set of related messages that make up a 


conversation between two Parties. The Party initiating a conversation determines the 
value of the ConversationId element that shall be reflected in all messages pertaining to 
that conversation. It remains constant for all messages within a conversation. 

• Compression - attribute (true/false) allows control compression of response. Switching to 
“true” value results in returning the result ZIPed Trip_SearchRS payload, so the response 
looks like ‘<Message>ZIPed Trip_SearchRQ payload</Message>’. 
• EchoToken - sequence number for additional message clarification assigned by the 
requesting host system. When a request message includes an EchoToken then 
corresponding response message MUST include an EchoToken with an identical value. 
• TimeStamp – indicates the creation date and time of the message in UTC using the 
following format specified by ISO 8601; YYYY-MM-DDThh:mm:ss with time values 
using the 24 hour clock (e.g. 20 November 2003, 1:59:38 pm UTC becomes 2003-11- 
20T13:59:38). 
• Target - used to indicate whether the request is for the Test or Production system. 
• Version - for all versioned messages the version of the message is indicated by a decimal 
value. 
• SequenceNmbr - used to identify the sequence number of the transaction as assigned by 
the sending system; allows for an application to process messages in a certain order or to 
request a resynchronization of messages if a system has been off-line and needs to 
retrieve messages that were missed. 
• PrimaryLangID - identifies the primary language preference for the form of travel 
represented in a collection. The human language is identified by ISO 639 codes. 
• AltLangID - identifies the primary language preference for the form of travel represented 
in a collection. The human language is identified by ISO 639 codes.

3 . 1 3 . O S I S e a r c h C r i t e r i a 

 

 

Read Request Type: 

ReservationReadRequest 

Elements: 

Osi 

Osi Elements: 

Text 

The user is able to request an XML response with the PNRs containing provided text describing 

Other Supplementary Information. Trip will compress (removes white spaces and non- 
alphanumeric) OsiCriteria/Text and use that to search against the values created from OSI Free Text 
by compressing it and reducing the length to the first 40 chars.

• Text non-alphanumeric values are removed for the search and when storing data. 


This search option is available only to AS customers.

To better serve you please note the following:

• To help ensure that our environment is free of viruses, our policy mandates that all messages 
received by Sabre from external sources follow special file name guidelines. File names must 
end in ".sabre.zip" or the zipped attachment will be removed by the email server (for 
example, “docs.zip” would need to be renamed to “docs.sabre.zip”). 
• If your correspondence is regarding a previously reported issue, please include the service 
incident (“SI”) number in the subject line of your message. 
• The subject line should contain a brief description of the problem or question. Please avoid a 
subject line containing only the words ‘Question’, ‘Issue’, ‘Problem’, etc. 
• In the body of the email include as much information as possible to explain the problem and 
the steps necessary to recreate it.
